## 3. Open parser questions (ﾂｧ7)

1. `pin task T()` -- DONE (commit c1eb731, 2026-06-20). The proto DECL
   form is now `val`-ONLY: `var`/`pin` + `task`/`func` are rejected
   (`var task T` / `pin task T` -> "expected <id>"). Same commit also
   dropped `var func` (only `val func` / bare `func` remain).
   1b. ASSIGN form `pin T = task(){}` (pinning a proto VALUE) -- distinct
   from the decl form above. RESOLVED: no static check (2026-06-22).
   Pinnability is a runtime property of the VALUE; the `pin` RHS can be
   a call return, table field, var, or conditional the parser cannot see
   through (`pin t = f()`, `pin t = tbl.x`, `pin t = T`). A static check
   would catch only literal `pin x = task(){}` / `func(){}` -- a narrow
   lint that misses every indirect case. The runtime `<close>`-on-non-
   closable check (today) is the ONLY sound+complete mechanism. Optional
   follow-up (-> ﾂｧ5): decide if `pin` is task-specific (meta_xtask/
   meta_tasks only) or general close-on-scope; if task-specific, upgrade
   the raw Lua "got a non-closable value" message to "invalid pin :
   expected task instance" in `atm_pin_chk_set`; if general, leave as-is.
2. `toggle T()` on a PROTOTYPE vs instances only -- RESOLVED: instances
   only, already enforced (2026-06-22). No static check (same reasoning
   as 1b: target is normally a var/return/field the parser can't type).
   Runtime already restricts to meta_xtask/meta_tasks: `toggle T(false)`
   on a proto -> "invalid toggle : expected task" (clean + source-
   attributed, unlike pin's raw message). Nothing to do.
3. cap `task` / non-cap `func` naming rule -- WONT DO (2026-06-20).
   Considered: force `task NAME` uppercase + `func NAME` lowercase so
   capitalization signals task-vs-func. Rejected: too opinionated;
   forbids PascalCase factory funcs (e.g. exs/clicks `func Buffer`/
   `func Debounce`). spawn-of-rawfunc is already runtime-rejected, so
   the convention is not needed for correctness.

DONE (commit c1eb731): `::`/`.` method+field form REJECTED for `task`
(`task X::m` / `task X.Y` -> "expected '('"); gated to `sub=='func'`,
so only `func M.f` / `func o::m` keep it. Manual (commit 6034316)

## 5. task-term `pin t =` double-pin -- RESOLVED: correct rejection (2026-06-22)

`tst/tasks.lua` "task-term 1/2/3" emit a trailing
`invalid assignment : expected unpinned value` at `pin t = spawn(...)()`.
This is NOT a bug: it is a LEGITIMATE double-pin, correctly rejected.

Mechanism (traced; `e` and `t` are the SAME xtask):
1. `spawn { pin e = await(true) ... }` starts task A, runs to await.
2. RHS `spawn (task(){})()` spawns B with an EMPTY body -> B terminates
   immediately.
3. B's termination broadcast wakes A's `await(true)` SYNCHRONOUSLY,
   mid-assignment.
4. A resumes `pin e = <B>` -> pins B (`B.pin=true`), prints `ok xtask:`.
5. line 5 `pin t = <B>` -> B already pinned -> error (correct).

Invariant: a task is pinned EXACTLY ONCE; later bindings must be `val`
(case `pin e` + `val t` is CLEAN; `val e` + `pin t` fails the other way
with "expected pinned value"). Per decision, double-pin SHOULD error.

Tests left as-is: trailing error is correct; `assertfx` (find) matches
the `ok xtask:` prefix, so they PASS. The shared-object surprise (an
empty task's termination waking `await(true)` and letting an unrelated
block pin it) is existing await/termination semantics, not a pin issue.

## Reference: no surface bless `task(f) -> :task`

`task` is a keyword, so `task(f)` reads `(f)` as the anon-proto PARAM
LIST then demands `{ body }` (`task(f)` -> "expected '{'"); no room to
also mean "bless f". Once the ﾂｧ4 gate lands, `xtask(f)` from a raw func
also fails -> NEITHER proto-from-func nor instance-from-func is surface-
expressible. Intentional: task-ness is DECLARED. Escapes: native
``\`task(f)\``` (works); compiler emits `task(fn)` directly (atm_behavior).
