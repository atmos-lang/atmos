# Plano 2: Sugar de await com `any` / `all`

**Data:** 2026-05-22
**Escopo:** Expor combinadores de padrões de await como funções públicas do runtime
**Não-escopo:** Mudanças no parser ou em `prim.lua`/`coder.lua`

## 1. Decisão

Adicionar duas funções públicas ao runtime — `any(...)` e `all(...)` — que constroem
padrões de await compostos. Não há mudança de parser nem de gramática. As funções são
identificadores comuns que entram no namespace global do Atmos, consumidos
naturalmente pela função `await` existente.

## 2. Motivação

Hoje, para combinar múltiplos awaits, o programador escreve:

```
par_or { await :X } with { await :Y }
par_and { await :X } with { await :Y }
```

Verboso para o caso degenerado de awaits singulares. Com `any`/`all`:

```
await any(:X, :Y)
await all(:X, :Y)
```

Quatro tokens em vez de dez. Sem perda de expressividade — combinações com
side-effects continuam usando `par_or`/`par_and`.

## 3. Implementação

### 3.1. Runtime (`lua-atmos/atmos/run.lua`)

As funções `M._or_` e `M._and_` já existem (linhas 684-714) e constroem exatamente
o descritor necessário: `{tag='_or_', ...}` e `{tag='_and_', ...}`. Adicionar:

```lua
M.any = M._or_
M.all = M._and_
```

Cerca de duas linhas, próximas às definições atuais. Os nomes `_or_` e `_and_` podem
permanecer como aliases internos para compatibilidade, ou serem renomeados — decisão
de estilo.

### 3.2. Despachador (`check_ret` em `run.lua` linhas 507-576)

Nada a mudar. O despachador já trata `tag='_or_'` e `tag='_and_'` recursivamente,
incluindo aninhamento arbitrário.

### 3.3. Parser/coder (`atmos/src/*.lua`)

Nada a mudar. `any` e `all` são identificadores comuns. `await any(:X, :Y)` parseia
hoje como `await(any(:X, :Y))` — uma chamada de função passada como argumento. O
parser não precisa saber de nada.

### 3.4. Namespace global

`any` e `all` precisam estar acessíveis sem prefixo, igual a `spawn`, `emit`, etc.
Verificar onde o runtime expõe os builtins ao namespace de Atmos
(provavelmente `init.lua` ou `global.lua`) e adicionar.

## 4. Semântica

`any(e1, e2, ..., en)` retorna um descritor que, quando passado a `await`, suspende
até que qualquer um dos `ei` seja satisfeito.

`all(e1, e2, ..., en)` retorna um descritor que, quando passado a `await`, suspende
até que todos os `ei` sejam satisfeitos. A ordem de satisfação não importa; o `await`
retorna quando o último for completado.

Os argumentos `ei` podem ser de qualquer tipo aceito por `await` (manual 5.9.3):

- tags (`:X`)
- valores literais
- tasks (`t`) ou pools de tasks (`ts`)
- clocks (`@5`)
- funções predicado (`\{it>10}`)
- booleanos (`true`/`false`)
- recursivamente, outros `any(...)` ou `all(...)`

### 4.1. Valor de retorno

- `any` retorna o payload do evento que satisfez (igual ao await singular do ramo
  vencedor).
- `all` retorna uma tabela com os payloads de cada `ei`, na ordem dos argumentos.

Comportamento idêntico a `par_or`/`par_and` em posição de expressão.

### 4.2. Aninhamento

```
await any(:X, all(:Y, :Z))
```

Acorda quando `:X` ocorre OU quando ambos `:Y` e `:Z` ocorreram. Composição livre
porque `check_ret` é recursivo.

### 4.3. Singletons

`any(:X)` e `all(:X)` são equivalentes a `await :X` direto. Permitir é mais simples
que proibir; útil para código gerado por macro/template.

## 5. Casos de uso

```
await any(:X, :Y)                          ;; primeiro entre :X ou :Y
await all(:X, :Y)                          ;; quando :X e :Y ambos ocorreram

await any(:click, @5)                      ;; click ou timeout = watching inline
await all(t1, t2, t3)                      ;; junção de 3 tasks = par_and de awaits

await any(:cancel, all(:input, :submit))   ;; cancel ganha; senão, input E submit
```

## 6. Relação com construtos existentes

| Hoje | Com `any`/`all` |
|---|---|
| `par_or { await :X } with { await :Y }` | `await any(:X, :Y)` |
| `par_and { await :X } with { await :Y }` | `await all(:X, :Y)` |
| `watching :X { body }` | `await any(:X, do { body; :__done })` (ver §8) |

Mantém `par_or`/`par_and` como a forma canônica quando os ramos têm side-effects
ou são blocos compostos. `any`/`all` é açúcar para o caso de awaits singulares como
expressão.

`watching` não é redutível diretamente porque envolve abort estrutural de um bloco,
não apenas espera; sua redução continua sendo via `par_or` conforme Plano 1 §4.3.

## 7. O que NÃO muda

- Sintaxe do `await`: continua aceitando `await(:X, :Y)` como matching multi-arg
  contra um único emit (manual 5.9.3 ponto "`...` | each of the arguments must match
  each of the `emit` arguments"). Não há conflito: `await(any(:X, :Y))` é um único
  argumento (a tabela construída por `any`); `await(:X, :Y)` continua sendo dois.
- Operadores `||` e `&&`: continuam sendo lógicos booleanos com semântica Lua.
- `par_or` / `par_and` / `watching`: continuam funcionando como hoje.

## 8. Diferenças entre `any`/`all` e `par_or`/`par_and`

**`any`/`all`** operam em padrões de await — descritores passivos. Não criam tasks.
Suspendem a task atual via `coroutine.yield` única. Eficientes para combinar awaits
singulares.

**`par_or`/`par_and`** operam em blocos com side-effects. Criam transparent tasks
para cada ramo. Pagam o custo de um spawn por ramo. Necessários quando cada ramo
executa código antes do await.

Regra de bolso a documentar:

> Use `any`/`all` quando cada ramo é apenas um await singular.
> Use `par_or`/`par_and` quando cada ramo executa código.

## 10. Itens em aberto

- **Nomes alternativos**: `any`/`all` é claro mas curto demais para alguns gostos.
  Alternativas: `await_any`/`await_all`, `or_`/`and_`, `first`/`every_of`. Decidir
  baseado em legibilidade real em código existente.
- **Aceitar tabela como atalho**: `any({:X, :Y, :Z})` poderia funcionar identicamente
  a `any(:X, :Y, :Z)` para listas geradas dinamicamente. Trivial de adicionar
  (checar se primeiro arg é tabela sem tag e fazer unpack), mas pode confundir com
  matching de tabela. Decidir.
- **Variante de bloco**: se a comunidade pedir, `await any { :X; :Y; :Z }` é uma
  extensão futura via parser change. Não fazer agora — esperar feedback.
- **Performance**: `any`/`all` deve ser comparável a `par_or`/`par_and` no caso de
  awaits singulares, ou até mais barato (não há spawn). Verificar com benchmark
  trivial.
