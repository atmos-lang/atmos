# Plan 2: Await sugar with `&&` / `||` / `!` operators

**Date:** 2026-05-22
**Scope:** Reinterpret `&&`, `||`, and `!` as await-pattern combinators
when they appear inside `await`, `watching`, and `every` argument lists.
No new identifiers, no parser table changes — only an AST post-process.
**Out of scope:** Changes to `coder.lua`, `lexer.lua`, or operator
precedence. Runtime additions are limited to a single helper for `!`.

## 1. Decision

Treat `&&`, `||`, and `!` as syntactic sugar for the await combinators
**only** in argument position of `await`, `watching`, and `every`.
Outside those contexts they remain ordinary logical operators with Lua
semantics. The disambiguation is purely contextual and happens at the
parser stage as an AST post-process inside the `await`/`watching`/`every`
branches of `prim.lua` — no parser-flag plumbing, no new keywords.

The compiler emits the lua-atmos **table format**:

```
{ 'or' , a, b, ... }
{ 'and', a, b, ... }
{ 'not', a }
```

— a plain positional array whose first element is the discriminator.
No new global identifiers are exposed to Atmos code. The runtime
recognises the format in `await_to_table` / `check_ret` (see §3.3).

## 2. Motivation

Today, to combine multiple awaits, you write:

```
par_or  { await :X } with { await :Y }
par_and { await :X } with { await :Y }
```

Verbose for the degenerate case of singleton awaits. With `&&` / `||`:

```
await :X || :Y
await :X && :Y
await :X && !:Y     ;; :X happens while :Y has not
```

Few tokens, reusing operators every Atmos user already knows. No new
identifiers in the global namespace.

## 3. Implementation

### 3.1. AST post-process in `src/prim.lua`

A single helper near the `await` branch walks an expression tree and
rewrites:

- `bin` nodes with `op.str` ∈ {`&&`, `||`} into Atmos `table` AST
  nodes with positional entries `[1]='or'|'and', [2]=lhs, [3]=rhs`;
- `uno` nodes with `op.str == '!'` into `table` AST nodes with
  positional entries `[1]='not', [2]=operand`.

The walk descends through:

- other `bin` `&&`/`||` nodes (recursive composition);
- `uno` `!` nodes (so `!(:X || :Y)` lowers);
- `parens` nodes (so `(:X || :Y) && :Z` lowers correctly).

It stops at any other node tag (calls, indexes, lambdas, literals, tags,
clocks, `acc`, etc.) — those keep their normal logical meaning.

```lua
local function await_ast_logical (e)
    local function combinator (name, args)
        local es = {
            {
                k = { tag='num', tk={tag='num', str='1'} },
                v = { tag='str', tk={tag='str', str=name} },
            },
        }
        for i, a in ipairs(args) do
            es[#es+1] = {
                k = { tag='num', tk={tag='num', str=tostring(i+1)} },
                v = await_ast_logical(a),
            }
        end
        return { tag='table', es=es }
    end
    if e.tag == 'parens' then
        return await_ast_logical(e.e)
    elseif e.tag == 'bin' and (e.op.str=='&&' or e.op.str=='||') then
        local name = (e.op.str=='&&') and 'and' or 'or'
        return combinator(name, { e.e1, e.e2 })
    elseif e.tag == 'uno' and e.op.str == '!' then
        return combinator('not', { e.e })
    else
        return e
    end
end
```

The Atmos `table` coder emits `atm_table{[1]="or", [2]=..., ...}`. The
`atm_table` runtime helper (`src/run.lua`) just sets the standard table
metatable — does not interfere with positional access.

### 3.2. Call sites in `src/prim.lua`

Three branches consume await patterns as call arguments. Each one
rewrites **only the first argument** through `await_ast_logical`. The
remaining arguments are **payload positions** matched against `emit(...)`
arguments and must not be transformed (see §3.2.1).

| construct | branch in `prim.lua` | what to rewrite |
|---|---|---|
| `await(...)` | generic `await` branch (line ~183) | `call.es[1]` only |
| `watching ... { ... }` | `watching` branch (line ~684) | `awt[1]` only |
| `every ... { ... }` / `every ... in ... { ... }` | `every` branch (line ~643) | `awt[1]` only |

### 3.2.1. Why only the first argument

`lua-atmos` treats the first argument as the pattern type (combinator,
clock, tag-match, task, function, boolean, ...) and the remaining
arguments as multi-arg payload positions matched against `emit(...)`.
The combinator branch of `await_to_table` ignores the trailing `...`
(run.lua line ~631) — only the first slot can carry the combinator.

Concretely:

```
await(:X, 10)         ;; multi-arg match: emit(:X, 10)
await(:X || :Y)       ;; combinator in slot 1, no payload
await(:X || :Y, 10)   ;; combinator in slot 1, 10 is payload (subject to runtime support)
await(10, :X || :Y)   ;; INVALID -- combinator in payload position, not rewritten
```

The compiler does not lift operators out of payload positions; that
keeps the rule simple and matches the runtime's existing dispatch
asymmetry.

The `await Id(args)` shorthand (line ~166) is the spawn-and-await form;
it does not take await patterns and is left untouched.

### 3.3. Runtime (`lua-atmos/atmos/run.lua`)

The runtime accepts the new positional-table format:

```
{ 'or' , a, b, ... }
{ 'and', a, b, ... }

```

Required changes:

- `await_to_table` (around line 623): add a branch detecting
  `type(e)=='table' and (e[1]=='or' or e[1]=='and' or e[1]=='not')`;
  build the internal descriptor and recursively process operands
  starting from index 2.
- `check_ret` (around line 507): add cases for `awt[1]=='or'`,
  `awt[1]=='and'`, `awt[1]=='not'` mirroring the existing `_or_`/`_and_`

The legacy `_or_`/`_and_` constructors and `tag='_or_'`/`tag='_and_'`
dispatch can remain for backwards compatibility, or be deprecated —
orthogonal decision.

Semantics of `{'not', p}`: satisfied iff `p` is **not** satisfied by the
current event. Only meaningful inside a composite (`&&` / `||`); a bare
`await !:X` would fire on the very first non-`:X` event. See §4.3.

## 4. Semantics

Inside `await(...)` / `watching(...)` / `every(...)` argument lists:

- `a || b` — suspend until **either** `a` or `b` is satisfied. Returns
  the payload of the winning side.
- `a && b` — suspend until **both** `a` and `b` are satisfied (in any
  order). Returns a table with both payloads in source order.
- `!a` — satisfied iff `a` is **not** satisfied by the current event.

Operands can be any value `await` already accepts (manual 5.9.3): tags,
literals, tasks, task pools, clocks, predicate functions, booleans, or
recursively other combinator expressions.

### 4.1. Return value

- `||` returns the payload of the winning branch.
- `&&` returns a table with the payloads of each operand, in source
  order.
- `!` returns the (negated) check result — useful only in composites.

Identical to `par_or` / `par_and` in expression position for the binary
cases.

### 4.2. Nesting & precedence

`&&` and `||` already live at the same precedence level in
`parser_5_bin`, and the parser already rejects mixing them without
parentheses:

```
operation error : use parentheses to disambiguate
```

So `await :X || :Y && :Z` is a parse error today and remains so under
this plan. Users must disambiguate:

```
await (:X || :Y) && :Z
await :X || (:Y && :Z)
```

`!` binds tighter than `&&`/`||` (it is a unary prefix in
`parser_4_pre`), so `await !:X && :Y` parses as `(!:X) && :Y`. Same as
Lua / C intuition.

### 4.3. Singletons & corner cases

- `await :X` is unchanged — no `bin`/`uno`, no rewrite.
- `await !:X` alone: fires on any event that does not match `:X`. Allowed
  but rarely useful; left in for compositional regularity.
- `await !:X && :Y`: typical use — wait for `:Y` only while `:X` has not
  intervened. Composes naturally with `watching`-like patterns.

## 5. Use cases

```
await :X || :Y                          ;; first of :X or :Y
await :X && :Y                          ;; when :X and :Y both occurred
await :X && !:Y                         ;; :X without :Y intervening

await :click || @5                      ;; click or timeout
await t1 && t2 && t3                    ;; join of 3 tasks

await :cancel || (:input && :submit)    ;; cancel wins; else input AND submit

watching :X || :Y { body }              ;; abort on either :X or :Y
watching :click || @10 { body }         ;; abort on click or after 10s

every :tick || :force { body }          ;; loop on either event
every :tick && !:pause { body }         ;; tick while not paused
```

## 6. Relation to existing constructs

| Today | With operators |
|---|---|
| `par_or  { await :X } with { await :Y }` | `await :X \|\| :Y` |
| `par_and { await :X } with { await :Y }` | `await :X && :Y` |
| `watching :X, :Y { body }` (implicit OR) | `watching :X \|\| :Y { body }` |
| (none — required `watching` wrap) | `await :X && !:Y` |

`par_or` / `par_and` remain canonical when each branch has side-effects
or compound bodies. Operators are sugar for the singleton-await case in
expression position.

`watching` and `every` currently accept a comma-separated list of awaits
with implicit OR semantics. The `||` form makes that OR explicit; `&&`
and `!` open compositions that the comma form cannot express.

## 7. What does NOT change

- `&&` / `||` / `!` outside `await`/`watching`/`every` argument lists:
  still logical operators with Lua semantics.
- `await(:X, :Y)` two-argument matching against a single emit (manual
  5.9.3 "each of the arguments must match each of the `emit`
  arguments"): still works. The rewrite happens **per argument**, not
  across arguments.
- `par_or` / `par_and` / `watching` block forms: untouched.
- `===`, `=!=`, `?>`, `!>`, `++`, etc.: untouched.

## 8. Contextual disambiguation

### 8.1. Bare `await` does not absorb combinator operators

The bare-await shorthand `await :X` accepts a single primary as its
argument. A following `&&` / `||` / `!` is **not** part of the await
call:

```
await :X || :Y     ;; compiles as (await :X) || :Y -- logical OR
```

The chain `parser_2_suf` → `parser_5_bin` builds `call(await, [:X])`
first, then `||` becomes a binary operator applied to the call result.
The `||` is outside the await argument list and the AST post-process
(§3.1) does not reach it. The resulting Lua is a normal logical OR on
the value returned by `await`.

This is treated as a **user logic error**, not enforced by the
compiler. The rule users must learn:

> Combinator operators (`&&`, `||`, `!`) inside `await` require the
> parenthesized form: `await(:X || :Y)`. The bare `await :X` form
> accepts a single primary only.

`watching` and `every` are unaffected: their arguments are parsed by
`parser_list(',', '{', parser)`, which already descends through
`parser_5_bin` before stopping at `{`, so `watching :X || :Y { ... }`
naturally lives in the await-arg list and is rewritten correctly.

### 8.2. Operators inside nested call args remain logical

Inside an await argument, a tree of `&&`/`||`/`!` nodes is rewritten;
everything else passes through. So:

```
await x && f(a || b)
```

becomes:

```
_and_(x, f(a || b))
```

The `||` inside `f(...)` is **not** rewritten because the walker stops
at `call` nodes — it does not descend into function arguments. This
preserves Lua semantics where the user clearly wants logical OR.

If the user wants the inner `||` to be a combinator too, they pass the
combinator-producing expression directly without wrapping:

```
await x && (a || b)    ;; both rewritten
```

## 9. Difference between operators and `par_or` / `par_and`

**`&&` / `||` / `!`** operate on await patterns — passive descriptors.
No tasks created. Suspend the current task via a single
`coroutine.yield`. Efficient for combining singleton awaits.

**`par_or` / `par_and`** operate on blocks with side-effects. Spawn a
transparent task per branch. Pay the cost of a spawn per branch.
Necessary when each branch runs code before the await.

Rule of thumb to document:

> Use `&&` / `||` / `!` when each branch is a single await pattern.
> Use `par_or` / `par_and` when each branch runs code.

#### 12.3.1 Patch `check_ret` (around line 513)

The current dispatcher matches on `awt.tag`. Add three new cases that
match on `awt[1]` (string discriminator). Place them at the top of the
function, before the existing `awt.tag == '_or_'` / `'_and_'` cases.

```lua
local function check_ret (awt, ...)
    -- awt = await pattern | ... = occurring event arguments
    local e = awt[1]
    local mta = getmetatable(awt)
    local mte = getmetatable(...)

    -- NEW: positional-table combinators
    if awt[1] == 'or' then
        for i = 2, #awt do
            local vs = { check_ret(awt[i], ...) }
            if vs[1] then
                return table.unpack(vs)
            end
        end
        return false
    elseif awt[1] == 'and' then
        local rets = {}
        for i = 2, #awt do
            local vs = { check_ret(awt[i], ...) }
            if not vs[1] then
                return false
            end
            rets[#rets+1] = (#vs>2 and {table.unpack(vs,2)}) or vs[2]
        end
        return true, table.unpack(rets)
    elseif awt[1] == 'not' then
        local vs = { check_ret(awt[2], ...) }
        return (not vs[1]), table.unpack(vs, 2)

    -- EXISTING: tag-based dispatch
    elseif awt.tag == '_or_' then
        ...
```

Note: `'and'` here is a "snapshot" check — every operand must match the
same event. For the "in any order, across events" semantics of the
existing `_and_` (which marks branches `_ok_` as they fire), use the
existing path or add a stateful variant later. Decide based on the
intended semantics for `&&` (see §4).

#### 12.3.2 Patch `await_to_table` (around line 631)

The current code dispatches on `e.tag == '_or_' or '_and_'`. Add a
branch above it that recognises the positional-table format:

```lua
local function await_to_table (e, ...)
    local T
    if type(e) == 'table' then
        if (getmetatable(e) == meta_task) or getmetatable(e) == meta_tasks then
            T = { tag='_==_', e,... }
        elseif S.is(e) then
            T = { tag='_==_', spawn(function() return e() end),... }

        -- NEW: positional-table combinators
        elseif e[1] == 'or' or e[1] == 'and' or e[1] == 'not' then
            T = e
            for i = 2, #T do
                T[i] = await_to_table(T[i])
            end

        elseif e.tag=='_or_' or e.tag=='_and_' then
            ...
```

The branch leaves slot 1 (`'or'`/`'and'`/`'not'`) alone and recurses
into slots 2..N so nested combinators lower correctly.
