# 260616-gte : complete deep comparisons `=>=` `=<=`

## Upstream (semantics)

Defined & tested in lua-atmos:
`/x/lua-atmos/atmos/.claude/plans/done/260616-x.md`

`atmos.x` exposes ONE primitive `gte(a,b)` ("a is a supertype of b").
`eq`/`is` derive from it; there is **NO `X.lte`**. So `=<=` is the
flipped `gte`:

- `===`  → `X.eq`            (= `gte(a,b) and gte(b,a)`)
- `=!=`  → `not X.eq`
- `=>=`  → `X.gte(a, b)`
- `=<=`  → `X.gte(b, a)`     (NOT `X.lte`)

## Context

Deep comparison currently has only equality:

- `===`  deep equal      → `X.eq`
- `=!=`  deep not-equal  → `(not X.eq)`

Complete the family with deep ordered comparisons:

- `=>=`  deep greater-or-equal  → `X.gte`
- `=<=`  deep less-or-equal     → `X.lte`

(assumed to exist in the lua-atmos `atmos.x` module, like `X.eq`;
bound as the uppercase global `X` in `src/exec.lua:23`)

These are the structural variants of the plain numeric `>=` / `<=`,
which already compile straight to Lua operators.

## Risks / open

- Naming RESOLVED: lua-atmos has only `X.gte`; `=<=` is `X.gte(b,a)`
  (no `X.lte`/`X.le`/`X.ge`). Verified in source `atmos/x.lua:31`
  and tests `tst/x.lua` (`--- GTE ---`).
- Lexing ambiguity: `=>` (if-expr arrow) and `>=` already exist.
  The greedy reader takes `=>=` only when the 3 chars are contiguous;
  `if c => x` keeps `=>` (space stops the run).
  Verify no source uses `=>=` meaning `=> =`.
- Semantics live in lua-atmos; this change is purely lex/parse/emit.
