# loop in/on — fold `every` into `loop`

## Decision

Fold `every` into `loop` as a source clause.
`in` iterates a collection; `on` awaits an event stream.
Drop the standalone `every` keyword.

```
loop : loop IDS? (in EXPR | on PATTERN)? BLOCK
```

## Forms

| form                  | meaning                  | replaces            |
| --------------------- | ------------------------ | ------------------- |
| `loop {}`             | infinite                 | —                   |
| `loop i {}`           | numeric                  | —                   |
| `loop ids in xs {}`   | data iterator            | —                   |
| `loop ids on :Y {}`   | event loop, bind event   | `every ids in :Y {}`|
| `loop on :Y {}`       | event loop, no binding   | `every :Y {}`       |

## Why in/on

- Symmetry: every form is `loop <vars> <source> {}`.
  A leading id is always a variable (loop-style commit), so there is no
  consume-then-disambiguate and no seed.
- `in`/`on` are a matched preposition pair: container (space) vs occurrence
  (time), which maps onto data-vs-events honestly.
- `on` is honest about the source domain and reads best in reactive code.
  We accept it over `await` (which is honest about the suspension mechanism)
  for ergonomics.

## `on` — the event-pattern marker

`on` means "upon event pattern", and is the dual of `in` (collections).
It is reserved to three sanctioned forms, all genuine "upon event" sites:

| form               | role                                  | spawns?           |
| ------------------ | ------------------------------------- | ----------------- |
| `loop on P {}`     | repeating handler (replaces `every`)  | no                |
| `spawn on P {}`    | one-shot concurrent handler           | yes (explicit)    |
| `toggle on P {}`   | gated subscription (recurring control)| (transparent)     |

Desugaring:

- `loop on P {}`  = the current `every(P, func(ids){block})` lowering.
- `spawn on P {}` = `spawn { await P; block }`.
- `toggle on P {}`= today's `toggle :P {block}` (tag form), with `on` fronting
  the controlling event.

Rule: **`spawn` is the only concurrency signal.**
`on`/`loop on` are blocking in the current task; concurrency is opt-in via
`spawn`.
A reader never wonders whether `on` quietly forked a task — without `spawn`,
control is sequential, like `await`.

Removed (each already has a direct spelling, so no keyword surface):

| removed              | use instead             |
| -------------------- | ----------------------- |
| `on P {}` (bare)     | `await P; ...`          |
| `spawn loop on P {}` | `spawn { loop on P {} }`|

`watching P {}` and `await P` keep their own keywords: `watching` is "until"
(a terminator, not a subscription), `await` is the primitive.

## What this removes

| removed                                      | because                          |
| -------------------------------------------- | -------------------------------- |
| `every` parser branch (`prim.lua`)           | folded into `loop`               |
| `parser_await` seed (node) base mode         | leading id is always a var now;  |
| (`await.lua`)                                | no consume-then-seed needed      |
| every's `err "expected identifier"` guard    | no var-vs-pattern fork           |
| loop/every asymmetry + var/pattern ambiguity | gone                             |

`parser_await` keeps its `prim` (juxtaposition) + pool + default modes for
`await(...)` and `watching`.

2. [DONE] parser `loop` (`prim.lua`):
    - parse optional `IDS` (existing `parser_ids`).

## Open

- `watching` / `await`: keep their own keywords. `watching` is "until" (a
  terminator), `await` is the primitive — neither is an "upon" subscription.
- multi-id on an event source: ALLOWED, no arity check. An event is a single
  value, so `loop a,b on :Y` binds `a`=event and extra vars (`b`) receive nil.
  Lowers to `every(:Y, \(a,b){...})`.
- keep `every` as a deprecated alias? Decision: no — clean removal.
- whether `spawn on` should also allow a binding (`spawn e on :Y {}`) like
  `loop e on :Y {}`, or stay binding-less.
