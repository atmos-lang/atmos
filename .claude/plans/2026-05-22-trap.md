# Plano 1: Throw como mecanismo raiz de abort

**Data:** 2026-05-22
**Escopo:** Reorganização semântica e documental do modelo de control flow do Atmos
**Não-escopo:** Mudanças no runtime `lua-atmos/atmos/run.lua` ou no compilador `atmos/src/*.lua`

## 1. Decisão

Apresentar `throw`/`catch` como o mecanismo raiz de transferência de controle não-local
no Atmos. Todos os outros construtos de abort/escape são documentados como **derivações
expansíveis** sobre um núcleo mínimo de 7 primitivas + um pré-requisito sintático.

Esta é uma reorganização conceitual e documental. O runtime continua usando os dois
mecanismos do Lua 5.4 que já usa hoje (`error`/`pcall` para intra-task,
`coroutine.close`/`<close>` para cross-task). A diferença é apenas como apresentamos o
modelo ao programador e como organizamos o manual.

## 2. Núcleo

**7 primitivas:** `loop`, `spawn`, `await`, `emit`, `throw`, `catch`, `toggle`

**1 pré-requisito sintático:** `do { ... }` — bloco com escopo léxico. Não tem semântica
própria além de agrupar; sua função é garantir que variáveis `<close>` (`pin`, `defer`,
tasks transparentes) disparem na ordem correta ao sair do escopo.

**Convenção de tags:** identificadores como `:__done`, `:__loop`, `:__func`, `:__do_:L`
são tags *gensym* — frescas a cada expansão, garantidamente sem colisão com tags de
usuário. O runtime atual já usa `'atm-loop'`, `'atm-func'`, `'atm-do'` para break/return/
escape; mantemos o padrão.

## 3. Justificativa de cada primitiva

| Primitiva | Por que é primitiva |
|---|---|
| `loop` | Recursão infinita estruturada com break/continue. Mapeia para AST node próprio porque `break`/`continue` precisam de catch instalado pelo `atm_loop`. |
| `spawn` | Cria task — coroutine + estrutura de árvore de tasks. Não há jeito de derivar em termos de mais simples. |
| `await` | Suspende a task atual via `coroutine.yield` esperando padrão. Todo bloqueio reativo passa por aqui. |
| `emit` | Acorda tasks suspensas via `coroutine.resume`. Único produtor de eventos. |
| `throw` | Transferência de controle não-local intra-task via `error()` com marker. |
| `catch` | Captura `throw` via `pcall` filtrando por tag. |
| `toggle` | Suspende/resume task preservando estado. Genuinamente irredutível (throw destrói estado; spawn-com-restart não recupera). |

`do { ... }` não conta como primitiva porque não introduz comportamento novo —
apenas delimita escopo para `<close>` automáticos.

## 4. Mapeamento de derivações

Todo construto que **não está no núcleo** deve ser apresentado no manual como
expansão sobre o núcleo. Lista exaustiva:

### 4.1. Control flow intra-task

```
break                ≡  throw :__loop                 (capturado por loop)
return v             ≡  throw :__func @{ret=v}        (capturado por func)
escape :L            ≡  throw :__do_:L                (capturado por do :L)
```

Tags `:__loop`, `:__func`, `:__do_:L` são privadas — `catch :Y { }` user-level
nunca as vê. Hoje o runtime usa as strings `'atm-loop'`, `'atm-func'`, `'atm-do'`
para o mesmo propósito.

### 4.2. Composições paralelas

```
par { A } with { B }
≡
do {
    spawn { A }
    spawn { B }
    await(false)
}
```

Spawns têm lifetime atado ao `do`. `await(false)` nunca acorda. Único modo de
terminar: abort externo (throw atravessando), que dispara o `<close>` do `do`.

```
par_or { A } with { B }
≡
catch :__done {
    par {
        do { A; throw :__done }
    } with {
        do { B; throw :__done }
    }
}
```

Vencedor lança `throw :__done`, atravessa fronteira de task até o catch enclosing,
e o `<close>` do `par` mata os ramos perdedores.

```
par_and { A } with { B }
≡
do {
    pin t1 = spawn { A }
    pin t2 = spawn { B }
    await t1
    await t2
}
```

Sem throw. Ordem dos awaits não importa: `check_task_ret` retorna imediatamente
para tasks já mortas.

### 4.3. Casos especiais com await singular

```
watching X { body }
≡
par_or { body } with { await X }
```

`watching` é apenas `par_or` onde um dos ramos é um await singular. Não merece
status próprio, apenas legibilidade sintática.

```
every X { body }
≡
loop {
    val v = await X
    body(v)
}
```

### 4.4. Combinadores de await (`_or_`, `_and_`)

São padrões consumidos por `check_ret`, não construtos próprios. Hoje aparecem
como `M._or_(...)` e `M._and_(...)`:

```
await _or_(:X, :Y)        ;; descritor {tag='_or_', ...} passado a check_ret
await _and_(:X, :Y)       ;; idem com tag='_and_'
```

Cobertos pelo Plano 2 (sugar de await).

### 4.5. Lifetime estrutural

```
defer { body }            ;; variável <close> implícita; __close roda body
pin x = expr              ;; variável <close>; __close roda abort/close conforme tipo
tasks T { ... }           ;; pool com meta_tasks que faz close recursivo
```

Cada um vira variável `<close>` no escopo `do { }` enclosing. `<close>` dispara em
ordem reversa de declaração, garantindo que destrutores aninhem corretamente.

## 5. Hierarquia de derivação

```
loop  spawn  await  emit  throw  catch  toggle           [núcleo]
  │     │      │     │      │      │
  │     └──────┼─────┼──────┘      │
  │            │     │             │
break/continue │     │           toggle X { }            [derivado direto]
  │            │     │
  │           par    │                                   [spawn + await(false)]
  │            │     │
  │       ┌────┴────┐│
  │       │         ││
  │     par_or    par_and                                [catch+throw / await seq]
  │       │
  │     watching                                         [par_or + await singular]
  │
every                                                    [loop + await]

return / escape                                          [throw com tag interna]

defer / pin / tasks                                      [<close> de Lua 5.4]
```

## 6. Apresentação no manual

A seção 5.9 ("Task Operations") deve ser reorganizada em três blocos:

**5.9.A. Primitivas** — uma subseção por primitiva do núcleo. Explicar a semântica
operacional direta, sem referência a derivações. Esta é a ordem canônica de leitura.

**5.9.B. Construtos derivados** — para cada construto não-primitivo (`par`, `par_or`,
`par_and`, `watching`, `every`, `break`, `return`, `escape`), incluir uma caixa:

```
> Expansão equivalente:
> 
>     <forma do construto>
>     ≡
>     <expansão em termos do núcleo>
```

Isso dá ao leitor uma compreensão precisa do significado sem precisar ler o runtime.

**5.9.C. Mecanismos de lifetime** — uma subseção sobre `<close>` (`defer`, `pin`,
`tasks`), separada porque o mecanismo é de Lua 5.4 e atravessa todas as primitivas.

## 7. O que NÃO muda

- O runtime continua usando `coroutine.close` para abortar tasks-filhas em
  `par_or`/`watching`/`pin out-of-scope`. As expansões documentadas são equivalentes
  *semanticamente*, não *operacionalmente*. Substituir a implementação seria o
  plano de opção A (rejeitada).
- A sintaxe atual não muda. `par_or { } with { }`, `watching X { }`, `every X { }`
  continuam existindo como açúcar sintático produzido pelo parser. Manual apenas
  documenta o que isso significa.
- O compilador continua emitindo chamadas para `M.par_or`, `M.par_and`, `M.watching`,
  `M.every`, etc. Nenhuma mudança em `prim.lua` ou `coder.lua`.

### 8.2. Comentários no código

- Em `lua-atmos/atmos/run.lua`, anotar cada função do runtime com `--@ primitive` ou
  `--@ derived: <expansão>`. Útil para leitor do código entender o que é estrutural
  vs conveniência.
- Em `atmos/src/prim.lua`, o coder já trata `par_or`/`watching`/`every` como
  chamadas de função normais (sem AST node próprio), o que já reflete o status
  derivado. Adicionar comentário-cabeçalho explicitando isso.

### 8.3. Mensagens de erro

Considerar: quando o usuário usa `watching` ou `par_or` mal, mensagens de erro
poderiam apontar para a expansão? Provavelmente não — adiciona ruído. Manter
mensagens em termos do construto que o usuário escreveu.

## 9. Itens em aberto

- **`async`**: não está na conversa. Provavelmente derivado de spawn + thread. Verificar.
- **`tasks` pool**: a expansão exata em termos de `<close>` precisa ser checada contra
  `meta_tasks` no runtime.
- **`if`/`ifs`/`match`**: não são control flow não-local, mas vale considerar se a
  apresentação como "primitiva vs derivado" se estende a eles ou se ficam fora.
- **Tags privadas vs nomes**: hoje o runtime usa strings (`'atm-loop'`). O manual
  pode documentar como tags (`:__loop`) para consistência conceitual, mesmo que a
  implementação use strings. Decidir.
